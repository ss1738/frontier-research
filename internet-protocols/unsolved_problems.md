# Internet Protocols & Network Infrastructure — Frontier Unsolved Problems

## TOP 3 GAPS

### 1. Post-Quantum PKI Migration — The Multi-Party Coordination Deadlock
- NIST finalized ML-KEM (FIPS 203), ML-DSA (FIPS 204), SLH-DSA (FIPS 205) in August 2024. US Executive Order June 2025: RSA/ECDSA deprecated for new deployments by 2030, disallowed entirely by 2035
- ML-DSA-87 X.509 certificate: ~10 KB vs ~1 KB for RSA-2048 — 10× size increase. For 3-level CA hierarchy: 30+ KB of certificate data per TLS handshake causing bandwidth and latency regressions in constrained environments (IoT, mobile, satellite)
- As of early 2024: only ~1.8% of TLS 1.3 connections to Cloudflare used post-quantum cryptography (hybrid X25519+Kyber)
- CA/Browser Forum Baseline Requirements not yet updated to allow ML-DSA in publicly-trusted certificates — expected 2026–2027
- IETF has not finalized OID and encoding standards for PQC in X.509. Without standardized OIDs, CAs cannot issue publicly-trusted PQ certificates
- Certificate revocation compounding problem: OCSP stapling responses and CRL files both carry ML-DSA signatures, making revocation infrastructure ~10× larger. No deployed fix for high-volume OCSP responders at scale
- **Specific gap**: The "dual-certificate" or "composite certificate" architecture (draft-ietf-lamps-pq-composite-sigs) has zero production implementation. The coordination problem is a multi-party prisoner's dilemma: CAs won't issue until browsers accept → browsers won't accept until CT logs support it → CT logs won't support until OIDs are final → OIDs aren't final until CA/B Forum votes. No single actor can unblock it alone. IETF draft-sheffer-tls-pqc-continuity-00 (downgrade protection) proposed but not adopted
- **Company unlock**: Certificate Transparency infrastructure operator that pre-deploys PQ-aware CT log shards + CA/B Forum proposal + browser vendor coordination breaks the prisoner's dilemma. Alternatively: FIPS 140-3 validated ML-DSA HSM vendor at scale (expected H2 2025, still not broadly available) = the single bottleneck for enterprise private PKI migration

### 2. BGP Security — RPKI/ROV Creates a New Attack Surface It Was Designed to Close
- BGP prefix hijacking remains endemic. 2024 incident: AS267613 (Brazilian ISP) hijacked Cloudflare's 1.1.1.1/32 and propagated through Tier-1 providers despite RPKI ROAs existing for that prefix — 300+ networks, 70 countries
- As of May 2024 (MANRS milestone): ~50% of global IP prefixes covered by Route Origin Authorizations (ROAs). All but one transit-free Tier-1 provider performs ROV. Tier-2/regional adoption highly uneven
- NEW attack vector (NDSS 2026, SHAMAN framework): with partial ROV deployment, an attacker announces an invalid route that propagates only through legacy non-ROV ASes — never reaching the victim, invisible to victim's monitoring. Overall hijacking success probability: 14.1%. Targeted attacks against specific high-value prefixes: 99.5% success
- ASPA (AS Provider Authorization for AS_PATH verification) still at IETF SIDROPS as draft-ietf-sidrops-aspa-verification revision 25. NIST released BRIO test tools August 2025. Zero production deployment
- SCION (clean-slate path-aware internet architecture): commercial deployment via Anapaya + partial IXP integration. Blocked because: (a) requires all on-path ASes to opt in, (b) transit ASes have no direct revenue incentive to run SCION in parallel with BGP, (c) non-trivial end-host software overhead. SCION in research for 15+ years; deployed in dozens of ISPs, not thousands
- **Specific gap**: No mechanism can cryptographically verify the full AS_PATH (not just origin) in a deployed, incentive-compatible way at global Internet scale. ASPA partially covers provider-customer relationships but cannot detect peer-to-peer route leaks or colluding ASes. arXiv:2507.14842 confirms data-plane verification gap — even if control-plane announces look valid, actual packet forwarding paths are unverified
- **Company unlock**: Commercial tunneled overlay cryptographically pinning AS paths for high-value traffic (financial, DNS root, CDN origin pulls) without requiring universal SCION adoption — SCION-as-a-VPN for critical infrastructure. Anapaya exists but targets sovereign networks. Financial services / DNS operator market for verified-path guarantees is unbounded

### 3. Inference-Aware Network Transport — TCP/IP Is Structurally Wrong for Disaggregated LLM Serving
- Disaggregated LLM inference (prefill-decode separation) became production architecture in 2025: Meta, llm-d (Red Hat/AWS, ~3,100 tokens/sec per B200 decode GPU), Kimi Prefill-as-a-Service (arXiv:2604.15039)
- The network bottleneck is KV cache transfer. For a 70B model with 32K context, KV cache is 10–50 GB per request batch. Current solutions use RDMA (InfiniBand or RoCE via UCX): KVDirect achieves 22+ GB/s, reduces per-request latency by up to 55%. TCP fallback degrades to ~10× worse throughput
- Cross-datacenter KV cache transfer over WAN does not work at today's KV cache sizes. Only becomes viable with next-gen hybrid-attention architectures (MLA in DeepSeek) that substantially compress KV cache
- In-network computation for LLM inference: arXiv:2603.28239 (ISA) shows up to 8.7× All-Reduce speedup for small messages, 3.8× for large, LLaMA-2 speedups demonstrated. P4-programmable FPGA SmartNICs running fixed-point ML inference emerging (arXiv:2507.00428). None target KV cache routing specifically
- Google engineers 2025 (SDxCentral): for inference at scale, network latency and memory bandwidth — not raw compute FLOPS — are the binding constraints. Memory bandwidth improved only 17× from 2012–2022 while 64-bit GPU FLOPS grew 80×
- LMCache (arXiv:2510.09665) and FlowKV (arXiv:2504.03775) are software-layer workarounds for shared prefix caching — running on top of protocols not designed for this
- **Specific gap**: No network transport protocol designed for the KV cache transfer access pattern: large (GB-scale), latency-sensitive (must complete within one decode iteration, ~50ms), content-identifiable (KV blocks keyed by model + layer + token-hash), and multicast-reusable (same prefix KV cache could serve thousands of concurrent requests). TCP/IP is optimized for point-to-point reliable streams; RDMA requires homogeneous fabric; neither supports KV-cache-native semantics (content-addressed, reuse-aware, multicast-capable)
- **Company unlock**: A KV-cache-aware network transport layer — SmartNIC/DPU firmware extension or new RDMA verb set — that understands content-addressed KV block identifiers, supports multicast delivery of shared prefix caches, and integrates with disaggregated inference schedulers. The network equivalent of what NVLink is to intra-node GPU communication, but for inter-node inference. No company has shipped this. Market: every hyperscaler running inference at scale

## WILD HYPOTHESIS
**NDN-Style Content-Addressed Routing as a Native KV Cache Fabric for LLM Inference**: Named Data Networking routes by content name — a request ("Interest") propagates through the network; any router that has the named content ("Data") can respond, with automatic multicast suppression and in-network caching. This architecture was designed for exactly the problem disaggregated LLM inference has: the same content (a KV cache block for a given model/layer/token-prefix hash) may exist on multiple nodes, should be served from the nearest copy, and should be cached automatically at intermediate points. The KV cache block identifier maps directly onto an NDN name hierarchy. Nobody has run this experiment. The 2025 NDN literature (arXiv:2505.04326, arXiv:2605.13104) focuses on IoT, CDN, and edge caching — not LLM inference. The first experiment: instrument a disaggregated vLLM cluster (4+ nodes, prefill + decode separated) with an NDN forwarding plane (NFD) as the transport for KV cache blocks. Measure: (1) cache hit rate for shared system prompts under concurrent request load, (2) total KV transfer bandwidth vs RDMA baseline, (3) Time-to-First-Token under cache-warm vs cache-cold conditions. The hypothesis: NDN's native multicast suppression and in-network caching would reduce KV transfer bandwidth by 40–70% for high-reuse workloads (customer service bots, RAG with shared corpora) where the same prefix is requested by thousands of concurrent users. If it holds: a new network primitive — Content-Addressed Inference Fabric (CAIF) — where KV blocks are routable objects in the network rather than private memory on a specific GPU.
